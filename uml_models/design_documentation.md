# CHMS UML Models - Design Documentation

## 1. Alert Generation System

[cite_start]The design of the Alert Generation System prioritizes the Single Responsibility Principle to ensure maintainability within the safety-critical context of the Cardiovascular Health Monitoring System (CHMS)[cite: 26, 40]. [cite_start]The subsystem is broken down into three primary components: `Alert`, `AlertGenerator`, and `AlertManager`[cite: 78, 79, 80]. [cite_start]The `Alert` class acts strictly as a data transfer object, holding the state, patient ID, and timestamp of the anomaly without executing any business logic[cite: 79]. 

[cite_start]The `AlertGenerator` is tasked exclusively with the real-time evaluation of patient data against defined thresholds[cite: 73, 78]. [cite_start]By referencing the `DataStorage` interface within `AlertGenerator`, we decouple the evaluation logic from the physical storage layer, making the system easier to test and modify[cite: 52]. [cite_start]Finally, the `AlertManager` handles the dispatching of triggered alerts to medical staff[cite: 80]. [cite_start]Separating the generation of alerts from their distribution ensures that if the hospital updates its notification protocols—such as switching from a dashboard notification to SMS—only the `AlertManager` requires modification[cite: 44, 48]. [cite_start]Furthermore, linking the system to an external `Patient` concept ensures that alert rules can eventually be personalized without fundamentally altering the core evaluation loop[cite: 70]. [cite_start]This modularity guarantees that adding new evaluation rules or alert types in the future will not disrupt the existing routing architecture[cite: 44].

## 2. Data Storage System

[cite_start]The Data Storage System is architected using a hierarchical, composition-based model to securely and efficiently manage incoming patient streams[cite: 82]. [cite_start]The core structure relies on `DataStorage`, `Patient`, and `PatientRecord`[cite: 94, 95]. [cite_start]`DataStorage` acts as the central repository, utilizing a Map to associate integer IDs directly with `Patient` objects[cite: 85]. [cite_start]This structure ensures fast, direct lookups of patients, which is critical for a system processing high-frequency real-time vitals[cite: 21]. 

[cite_start]Each `Patient` object encapsulates its own list of `PatientRecord` instances, strictly grouping time-stamped physiological data by the individual[cite: 95]. [cite_start]To prevent the main storage class from becoming bloated with complex querying logic, the `DataRetriever` class was introduced[cite: 48, 96]. [cite_start]This separation of concerns dictates that `DataStorage` focuses solely on writing and holding data, while `DataRetriever` handles the specific logic for filtering historical trends or extracting data windows for medical staff[cite: 83]. [cite_start]By encapsulating the patient list within private attributes, the system maintains strict privacy; data cannot be inadvertently exposed or modified by unrelated subsystems[cite: 54, 55]. [cite_start]This design natively supports future enhancements, such as implementing automated data deletion policies or granular access controls, by restricting direct access to the underlying data structures[cite: 87, 91, 92].

## 3. Patient Identification System

[cite_start]The Patient Identification System acts as a crucial security and validation barrier between the incoming simulator data and the core system logic[cite: 99, 101]. [cite_start]It is designed around the `HospitalPatient`, `PatientIdentifier`, and `IdentityManager` classes[cite: 108, 109, 110]. [cite_start]The `HospitalPatient` class encapsulates the sensitive, real-world data of the patient (such as medical history and real names) alongside the simulator's ID[cite: 54, 109]. 

[cite_start]To manage these records securely, the `IdentityManager` oversees the system's integrity[cite: 110]. [cite_start]It holds the registry of patients and provides dedicated methods to handle edge cases, such as when incoming data fails to match any known patient record[cite: 102, 106]. [cite_start]The `PatientIdentifier` acts as the public-facing interface for this subsystem[cite: 108]. [cite_start]When the system receives a data point, it delegates the matching logic to the `PatientIdentifier`, which in turn queries the `IdentityManager`[cite: 104]. [cite_start]This abstracted layer prevents other components, like the Data Access Layer or Alert System, from directly interacting with the hospital database[cite: 55]. [cite_start]This isolation is critical for privacy compliance; it restricts access to sensitive medical records strictly on a need-to-know basis[cite: 55]. [cite_start]If the hospital migrates to a different database backend in the future, only the internal logic of the `IdentityManager` will require updating, leaving the rest of the CHMS unaffected[cite: 51].

## 4. Data Access Layer

[cite_start]The Data Access Layer is designed with maximum extensibility in mind, heavily utilizing abstraction to shield the core system from the complexities of network protocols[cite: 115, 117]. [cite_start]The architecture centers on the `DataListener` interface, which provides a standard contract for various input methods[cite: 116, 123]. [cite_start]By implementing subclasses like `TCPDataListener`, `WebSocketDataListener`, and `FileDataListener`, the system can effortlessly swap or add new data sources without rewriting existing infrastructure[cite: 51, 52, 124]. 

[cite_start]Once raw data is received, it is completely separated from the storage mechanisms to enforce the Single Responsibility Principle[cite: 47, 48]. [cite_start]The `DataSourceAdapter` acts as a structural bridge, fulfilling the requirements of the internal `DataReader` interface while relying on a dedicated `DataParser` to standardize incoming strings into usable system objects[cite: 120, 125, 126]. [cite_start]This ensures that parsing logic for formats like JSON or CSV does not pollute the network listener classes[cite: 120, 121]. [cite_start]Ultimately, this modular design allows the external world (the simulator) to interact with the system seamlessly[cite: 115]. [cite_start]The core CHMS remains blissfully unaware of whether data is streaming via a live WebSocket connection or being read sequentially from a local log file, ensuring high adaptability[cite: 112, 117].
